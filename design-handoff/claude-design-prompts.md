# Claude Design — Layered Prompts (Deliverable 53)

Ten prompts to drive the ACMP UI design, from system to screens to audit. **Use them in order.** Feed `claude-design-input-package.md` alongside each prompt (it is the shared context). The prompts give structure and constraints but deliberately **leave visual creativity to Claude Design** — they specify *what must work*, not *what it must look like*.

**Global constraints to repeat in every prompt:** bilingual **English + Arabic with full RTL**; **light + dark**; **WCAG 2.2 AA**; **desktop + tablet**; **government-grade, calm, serious, information-dense** tone; accessible drag-and-drop with keyboard alternatives; Gregorian dates; sensitive internal tool (no marketing flair, no dark patterns).

**How to use:** run prompt 1 to set the product frame, prompt 2 to establish the design system (everything later reuses it), then 3 for navigation, then 4–8 for the screen clusters, then 9 (Arabic/RTL validation) and 10 (consistency audit). Iterate within each before moving on. Keep the design system from prompt 2 as the single source of truth that every later prompt extends.

---

## Prompt 1 — Product-wide design
> You are designing **ACMP**, the system of record for a government organization's Architecture Committee (topic → backlog → agenda → meeting → minutes → vote → decision → ADR → action → risk → dependency, with full traceability). Read the attached input package. Establish the **overall product design direction**: a calm, serious, trustworthy, information-dense interface for busy senior technical leaders, used on desktop and tablet. Define the high-level layout shell (top bar with global search, language toggle EN/AR, theme toggle, notifications, profile; role-filtered left navigation; main content; consistent artifact-detail pattern with a relationships panel). It must work in English and **Arabic (full RTL)**, light and dark, and meet WCAG 2.2 AA. Deliver: the app shell, the visual tone, and 2–3 representative screens (a role dashboard, the backlog, a topic detail) to set the direction. Do not over-style; prioritize clarity, density, and consistency.

## Prompt 2 — Design system
> Using the attached input package and the direction from prompt 1, create the **ACMP design system**. Define design tokens (color incl. a semantic **status color map** aligned to our status models, spacing, radius, elevation, motion) and **typography for both Latin and Arabic** (a proper Arabic typeface with correct shaping). Build the core component library: buttons, inputs, selects, multi-selects, date pickers, chips/badges (status + urgency + aging), tables (dense, sortable, filterable), cards, tabs, dialogs, toasts, navigation, breadcrumbs, the relationship panel, kanban card, calendar cell, timeline item, voting panel, and a rich-text editor. **Every component must support light/dark and full RTL** out of the box, and meet WCAG 2.2 AA (focus, contrast, labels, target size). Provide usage do/don'ts. This system is the single source of truth for all later screens.

## Prompt 3 — Sitemap & navigation
> Using the design system, design ACMP's **navigation and information architecture** per the input package's sitemap. Create the role-filtered left navigation (what each role — Chairman, Secretary, Member, Reviewer, Auditor, Administrator, Submitter, Guest — sees), the top bar, breadcrumbs, deep-linking, and the consistent **"Related / Go to" traceability affordance** present on every artifact detail page. Show the navigation in **both LTR and RTL** (fully mirrored). Demonstrate the empty/loading/error/**permission-denied** states for a representative protected route. Keep navigation shallow and stable.

## Prompt 4 — Dashboards
> Design ACMP's **dashboards** (input package §10.1 and `/docs/27`). Create **role-specific home dashboards**: Secretary (backlog health, agenda readiness, overdue actions, pending approvals), Chairman (decisions awaiting approval, upcoming meeting, open votes), Member (my topics, my actions, my votes). Then design the **Reports** gallery with interactive filters, drill-down, export, and executive/committee/stream/audit views, covering backlog status, topic aging, decision status, risk exposure, dependencies, throughput, and attendance. Charts must be readable in dark mode and RTL. Show empty and loading (skeleton) states. Favor scannable, drill-downable cards over decorative charts.

## Prompt 5 — Topic & backlog workflow
> Design the **backlog and topic** experience (input package §10.2–10.3). The backlog is the workhorse: design **list, table (dense, sortable, filterable, with saved views), kanban (status columns), calendar, and timeline** views, with aging/urgency indicators, owner, stream tags, and status chips. Kanban and prioritization use **drag-and-drop with a fully keyboard-accessible alternative** (move via menu/keys, with screen-reader announcements). Design **topic submission** (a multi-section form with type, justification, affected streams/systems, attachments/diagrams, urgency — with autosave and an unsaved-changes guard) and the **topic detail** page (header, sections, comments, and the relationships/traceability panel). Include empty/loading/error states. Show key screens in RTL.

## Prompt 6 — Meeting & agenda workflow
> Design the **agenda and meeting** experience (input package §10.4–10.6). The **agenda builder**: a two-pane layout to pull backlog topics into an ordered agenda with drag-and-drop ordering (+ keyboard alternative), per-item time-box, presenter assignment, a total-time-vs-meeting-length indicator, and publish+notify. The **meeting workspace**: agenda items as the spine, attendance capture, per-item discussion notes, and inline quick-create of decisions/actions, with live time tracking. The **minutes editor**: structured MoM with rich text (EN/AR), versioning, review→approve, and a clear **locked/immutable** state once published. Include all four states and RTL.

## Prompt 7 — Decision, voting & ADR workflow
> Design the **decision, voting, and ADR** experience (input package §10.7–10.9). The **voting panel**: eligible voters, options, a live tally, a **quorum indicator**, **attributed** votes (clearly showing who voted what — voting is always attributed, not anonymous), abstentions, vote comments, and the **chairman approval/override recorded by name**; a clear **closed/locked** state after the vote. The **decision record**: outcome (from our canonical set), rationale, alternatives, conditions, approving authority, effective date, affected systems, "convert to ADR", and a supersede flow — with immutable history. The **ADR detail**: MADR layout with status badge and supersedes/superseded-by links. Treat voting and decisions as deliberate, confirmed actions (no casual drag-and-drop). Show RTL and all states.

## Prompt 8 — Traceability & dependency visualization
> Design ACMP's **traceability and dependency** experience (input package §16) — the hardest UX problem: make relationships **understandable, not overwhelming**. Design (a) the compact **relationships panel** that appears on every artifact detail (grouped by relation type, showing both directions, with "go to"), and (b) an optional **dependency/impact graph view** with depth control, blocked-work and cross-stream-impact highlighting, and an **accessible list/breadcrumb fallback** for the graph. Default to shallow, expand-on-demand. Ensure the graph and its fallback both work in RTL and dark mode, and are keyboard navigable.

## Prompt 9 — Arabic & RTL validation
> Review the entire ACMP design for **Arabic and RTL correctness**. For every screen produced so far, verify: full layout mirroring (navigation, tables, kanban, forms, drag-and-drop, graphs, icons that are directional), correct Arabic typography and shaping, **consistent EN↔AR terminology** (use the shared glossary), Gregorian dates with localized formatting, and no truncation/overflow with longer Arabic strings. Flag anything that breaks in RTL or reads awkwardly in Arabic, and provide corrected versions. Treat Arabic as a first-class language, equal to English — not a mirrored afterthought.

## Prompt 10 — Final design-review & consistency audit
> Perform a **final consistency and quality audit** of the complete ACMP design. Check: every screen uses the prompt-2 design system (no one-off components or colors); status colors match our canonical status models everywhere; all four states (empty/loading/error/permission-denied) exist for every screen; **WCAG 2.2 AA** is met (contrast, focus, keyboard, labels, target size, no color-only status); light/dark and **LTR/RTL** are both complete; drag-and-drop always has a keyboard alternative; forms have autosave + unsaved-changes guards; the tone stays calm, serious, and dense throughout. Produce a prioritized list of inconsistencies and fixes, and deliver the corrected, final design system + screen set ready for engineering handoff.
