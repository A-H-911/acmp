# 41 — RAID Register (Risks · Assumptions · Constraints · Dependencies)

**Purpose:** Central register of all Risks, Assumptions, Constraints, and Dependencies for the ACMP program. Covers Deliverable 49.

**Status values:** Open · Monitoring · Mitigated · Closed · Accepted.
**Likelihood/Impact:** H = High · M = Medium · L = Low.
**Exposure:** H×H = Critical · H×M or M×H = High · M×M or H×L = Medium · L×any = Low.

## 2. Assumptions

| ASM-ID | Assumption | Rationale / Consequence if False | Owner |
|---|---|---|---|
| ASM-001 | ~~The org's Keycloak instance will be accessible to ACMP from the deployment VM at all times; ACMP does not need to manage a separate IdP.~~ **RESOLVED — FALSE (ADR-0015):** the org has no Keycloak. Mitigation realized: Keycloak is now provisioned as a bundled service in the ACMP Compose stack with an ACMP-owned realm (the "scope expansion" path was taken). | If false: ACMP needs an internal fallback login path or Keycloak must be provisioned as part of the Compose stack — a scope expansion. **This is now the adopted design (ADR-0015).** | Tech Lead |
| ASM-002 | Keycloak group/realm-role claims can be configured to carry ACMP's canonical role names (or a mapping can be agreed) without code change to Keycloak itself. | If false: ACMP must support a custom claim-mapping configuration layer (already planned in BL-019) or Keycloak admin deploys a custom OIDC mapper. | Tech Lead / Keycloak Admin |
| ASM-003 | All committee users (≤20) have access to the deployment VM's network (internal network or VPN); no public internet exposure required. | If false: TLS termination + network exposure policy must be designed and approved, widening the security perimeter review. | Secretary |
| ASM-004 | SQL Server will be provisioned on the same network segment as the ACMP Docker Compose stack, with a connection string accessible via environment variable. ACMP owns its own SQL Server instance and schema. | If false: shared SQL Server requires schema isolation and resource governance configuration, adding operational risk. | Tech Lead |
| ASM-005 | MinIO can be provisioned as a Docker container in the ACMP Compose stack (or as a separately provisioned S3-compatible endpoint) with adequate storage volume mounted from the host. | If false: a different S3-compatible provider must be configured behind `IFileStore`; no code change required (abstraction layer), but ops setup changes. | Tech Lead |
| ASM-006 | The committee operates a single committee with a stable, known membership; the canonical roles (Chairman, Secretary, Member, etc.) map cleanly to real people. There will be no need to model sub-committees or working groups in v1. | If false: multi-committee model is a significant scope expansion (OOS, explicitly excluded by settled decision). | Secretary / Chairman |
| ASM-007 | Dates are Gregorian only; no Hijri calendar support is required in v1 (settled decision, README §A). | If false: Hijri calendar is a Phase 3+ item requiring a calendar library and locale configuration that is not currently planned. | Secretary |
| ASM-008 | The platform will be used by ≤20 total users; no horizontal scaling, no HA cluster, no heavy performance engineering is required. 99.9% availability is achievable via a simple standby VM + nightly backup (settled, README §A). | If false: a load balancer, multiple app instances, and session-state externalization are needed — an architectural change. | Tech Lead |
| ASM-009 | The organization does not require a public-facing portal or external partner access in v1; ACMP is an internal-only tool. | If false: a separate security-reviewed external-access layer is required (OOS-12). | Chairman |
| ASM-010 | The Secretary is the primary product owner and has authority to make functional scope decisions during delivery without escalating each to a governance board. | If false: decision latency will slow sprints; a lightweight approval process must be defined. | Secretary |
| ASM-011 | The urgency SLA thresholds (Critical: 3 days in status, Urgent: 7 days, Normal: 21 days) are acceptable starting values; the committee will adjust them via configuration after PH-1 pilot. [Marked `[unverified]` in FR-038; must be validated with committee.] | If false: SLA thresholds must be redesigned before PH-1 go-live; aging indicator calibration changes. | Secretary |
| ASM-012 | The committee is willing to adopt the canonical decision outcome vocabulary (11 outcomes in README §E) as-is, possibly with bilingual labels; no additional custom outcomes are required. | If false: the outcome enum is expanded; no code architecture change but schema + UI changes required. | Chairman |
| ASM-013 | Webex is the only external meeting/collaboration platform in use by the committee; no Teams or Zoom integration is needed. | If false: a second meeting-platform adapter must be designed, widening Phase 2 scope. | Secretary |
| ASM-014 | The AC-029 downstream-link gate (P8d, OQ-045) proxies "≥1 downstream artifact" as **≥1 ActionItem sourced from the decision** (`SourceType=Decision, SourceId`), and **exempts the supersession successor** from the gate. Rationale: the Risk module + typed-edge Traceability (ADR-0008) land in P10, so Actions is the only linkable artifact today; and a supersession successor is drafted+issued atomically with zero links, so gating it would deadlock every correction (the successor corrects a prior that already passed the gate). `SourceId` is trusted input — no cross-module existence check. | If false (e.g. a decision must link a Risk with no Action, or supersession must itself re-satisfy the gate): the gate predicate widens when P10 adds Risk/typed edges, and the supersession path would need its own link-carry-forward or a draft-then-issue split. No schema change; the `IActionLinkDirectory` predicate + handler check are the only touch-points. | Tech Lead / Architect |
| ASM-015 | **P10c Traceability backbone assumptions.** (1) *No physical Artifact registry* — relationship edges self-describe via value snapshots `(Type, Id, Key, Title)` on both ends; matches docs/30 §2.1 SQL and reconciles ADR-0008's "registry" prescription via **ADR-0019** (amends 0008). (2) *AC-029 widened to a **curated** downstream set* — an edge satisfies the gate iff the decision is source of `recorded-as`/`resolves` or target of `implements`; upstream/lineage edges (`decided-by`/`derived-from`/`supersedes`) are excluded so a decision cannot be self-satisfied by its own topic. Superset of the Action-only gate → AC-029 stays Met. (3) *Title is a create-time snapshot* — may go stale on rename (accepted; deep-link fetches live data; keys never go stale). (4) *Owner-AiO edge-create deferred* — v1 create is Chairman/Secretary RBAC only (no ABAC handler for arbitrary artifact types; no UI). (5) *Panel shows explicit edges only* — projection of existing soft-refs (Decision→Topic, Action→source, Risk→subject) into edges is a later slice, and when built should be **read-time contract composition, not write-through** (one source of truth; keeps the upstream `decided-by` edge out of the AC-029 predicate). (6) *Edge endpoint types are NOT validated against the docs/30 §2.2 per-RelType matrix* — the create validator checks `IsInEnum` + source≠target only, not that (e.g.) `RecordedAs` runs Decision→ADR. Trusted-actor input (Chairman/Secretary only, mirrors ASM-014's `SourceId` posture). **Consequence:** a malformed edge (e.g. `Decision --RecordedAs--> Meeting`) can satisfy the AC-029 gate, since `DecisionHasDownstreamEdgeAsync` matches on source-type + RelType without checking the far-endpoint type. Acceptable at ≤20 on-prem users; the full 16-RelType type matrix is deferred. | If false: (1) a registry + per-module edge-writes would be needed (ADR-0001 cost); (2) the downstream RelType set is re-curated; (3) a read-time title-resolution seam is added; (4) an artifact-generic ABAC resource handler is built; (5) projection is added as its own slice; (6) the §2.2 per-RelType endpoint-type matrix is added to the validator. All are additive — no rework of the P10c edge schema or the Shared `ITraceabilityLinks` port. | Tech Lead / Architect |

## 3. Constraints

| CON-ID | Constraint | Source | Implication |
|---|---|---|---|
| CON-001 | **Self-contained deployment**: ACMP must not depend on the org's shared runtime infrastructure (no org Hangfire, ELK, Seq, or notification platform). ACMP bundles its own background processing, observability, and notification channels in its own Docker containers. | README §A; secretary directive 2026-06-24 | Every service the platform needs is in the Docker Compose stack — all runtime dependencies (incl. self-hosted Keycloak + SQL Server) are bundled per ADR-0015; the only external dependency is Webex (Phase 2). No shared runtime services may be referenced in configuration. |
| CON-002 | **Mandated technology stack**: Backend must be .NET 8 (LTS) / ASP.NET Core. Frontend must be React 18 + TypeScript + Vite. Database must be Microsoft SQL Server. No deviations without an explicit ADR. | README §A; ADR-0001–ADR-0003, ADR-0012 | No Go, Python (except Tarseem sidecar), Java, Postgres, MongoDB, or other stacks for the ACMP application itself. |
| CON-003 | **On-prem VM + Docker Compose; no Kubernetes**: Deployment is Docker Compose on one or two VMs. No Kubernetes, Helm, service mesh, or container orchestration platform. | README §A; ADR-0001 | Resource limits, secret management, and scaling are all VM/Compose-level concerns. No K8s features (ConfigMaps, Secrets, Ingress controllers) may be assumed. |
| CON-004 | **Scale ceiling ≤20 total users; right-size everything**: No HA cluster, no horizontal scaling, no heavy performance engineering. 99.9% availability via simple standby VM + nightly backups. | README §A (settled decision 2026-06-24) | Do not engineer for 1,000 concurrent users. Reject any architectural addition (Redis, message broker, CDN) whose sole justification is scalability beyond 20 users. |
| CON-005 | **No message brokers in ACMP's own infrastructure**: No RabbitMQ, Kafka, or similar broker as part of ACMP's own runtime. Background work is handled by app-owned Hangfire on SQL. | README §A; ADR-0001 | Hangfire + SQL outbox is the messaging layer. Async patterns are task-queue only, not event streaming. |
| CON-006 | **No email in v1**: Email notifications are deferred until an SMTP relay is available. In-app notification center is the sole channel in PH-1. | README §A (settled 2026-06-24); ADR-0005 | `INotificationChannel` abstraction must be in place so email adapter can be added without modifying dispatch code. Alert users only via in-app center and (PH-2) Webex. |
| CON-007 | **No public self-registration or anonymous access**: All access is provisioned by Administrator/Secretary via Keycloak. No public-facing registration endpoint. | README §A; ADR-0004; CON-003 | `POST /api/register` does not exist. OIDC callback is the only entry point. |
| CON-008 | **EN and AR are both first-class, not additive**: Full bilingual support and RTL layout from the first PH-1 release. Not a v2 retrofit. | README §A; ADR-0012; P-07 | Every UI component must be built with logical CSS from day 1. Every user-visible string must exist in both locale files before the feature ships. RTL regression must pass in CI. |
| CON-009 | **Keycloak is the sole IdP; roles sourced from Keycloak claims**: ACMP does not store or assign global roles. It reads claims on token validation and maps to canonical roles. | README §A; ADR-0004 | ACMP cannot have a "role management" screen that changes Keycloak data. Role assignment is done in Keycloak; ACMP reads the result. |
| CON-010 | **Single committee; no multi-committee generalization**: v1 is built for exactly one committee. No tenant IDs, no committee selection, no committee-scoped routing. | README §A (settled 2026-06-24) | Reject any data-model change that adds a committee entity as a root tenant discriminator unless a new explicit decision reverses this. |
| CON-011 | **Voting always attributed in v1; no anonymous voting**: Every voter's choice is recorded and visible to Auditor and Chairman. Anonymous voting mode is out of scope for v1. | README §A (ADR-0010; settled 2026-06-24) | The `anonymity` flag on the vote configuration (FR-070 references it) is an informational field only in v1; the system always records individual votes. Full anonymity support is a future decision. |
| CON-012 | **Gregorian dates only; no Hijri in v1**: All date storage, display, and SLA calculations use Gregorian calendar. | README §A (settled 2026-06-24) | No Hijri calendar library, no dual-calendar display, no Hijri date input. Localized Gregorian formatting (right-to-left digit order in AR) is acceptable. |
| CON-013 | **Keystone is optional**: The Research module must work fully standalone (manual entry). Keystone import is an enhancement. No hard dependency on Keystone availability. | README §A; ADR-0007 | Research Mission record can be created without a Keystone package reference. The Keystone import tool is a PH-2 feature, not a gate for the Research module. |
| CON-014 | **Webex is Phase 2, adapter-only**: No Webex calls in PH-1 code. All Webex integration is behind the `INotificationChannel` and `IWebexRecordingAdapter` abstractions. | README §A (settled 2026-06-24); ADR-0005 | PH-1 code must compile and run without Webex credentials configured. The adapter registers as no-op when Webex config is absent. |
| CON-015 | **AI extraction is Phase 3; manual always available**: No AI features in PH-1 or PH-2. When delivered in PH-3, AI extraction is off by default and requires Admin activation with data-residency confirmation. | README §A (settled 2026-06-24) | No LLM client code in PH-1 or PH-2 production code. Phase 3 LLM integration is behind a feature flag. |
