# Constraint Register — ACMP

| ID | Constraint | Source | Type | Status |
|---|---|---|---|---|
| CON-001 | **Self-contained deployment**: ACMP must not depend on the org's shared runtime infrastructure (no org Hangfire, ELK, Seq, or notification platform). ACMP bundles its own background processing, observability, and notification channels in its own Docker containers. (Implication: Every service the platform needs is in the Docker Compose stack — all runtime dependencies (incl. self-hosted Keycloak + SQL Server) are bundled per ADR-0015; the only external dependency is Webex (Phase 2). No shared runtime services may be referenced in configuration.) | README §A; secretary directive 2026-06-24 | Deployment | Approved |
| CON-002 | **Mandated technology stack**: Backend must be .NET 8 (LTS) / ASP.NET Core. Frontend must be React 18 + TypeScript + Vite. Database must be Microsoft SQL Server. No deviations without an explicit ADR. (Implication: No Go, Python (except Tarseem sidecar), Java, Postgres, MongoDB, or other stacks for the ACMP application itself.) | README §A; ADR-0001–ADR-0003, ADR-0012 | Technical | Approved |
| CON-003 | **On-prem VM + Docker Compose; no Kubernetes**: Deployment is Docker Compose on one or two VMs. No Kubernetes, Helm, service mesh, or container orchestration platform. (Implication: Resource limits, secret management, and scaling are all VM/Compose-level concerns. No K8s features (ConfigMaps, Secrets, Ingress controllers) may be assumed.) | README §A; ADR-0001 | Deployment | Approved |
| CON-004 | **Scale ceiling ≤20 total users; right-size everything**: No HA cluster, no horizontal scaling, no heavy performance engineering. 99.9% availability via simple standby VM + nightly backups. (Implication: Do not engineer for 1,000 concurrent users. Reject any architectural addition (Redis, message broker, CDN) whose sole justification is scalability beyond 20 users.) | README §A (settled decision 2026-06-24) | Technical | Approved |
| CON-005 | **No message brokers in ACMP's own infrastructure**: No RabbitMQ, Kafka, or similar broker as part of ACMP's own runtime. Background work is handled by app-owned Hangfire on SQL. (Implication: Hangfire + SQL outbox is the messaging layer. Async patterns are task-queue only, not event streaming.) | README §A; ADR-0001 | Technical | Approved |
| CON-006 | **No email in v1**: Email notifications are deferred until an SMTP relay is available. In-app notification center is the sole channel in PH-1. (Implication: `INotificationChannel` abstraction must be in place so email adapter can be added without modifying dispatch code. Alert users only via in-app center and (PH-2) Webex.) | README §A (settled 2026-06-24); ADR-0005 | Product | Approved |
| CON-007 | **No public self-registration or anonymous access**: All access is provisioned by Administrator/Secretary via Keycloak. No public-facing registration endpoint. (Implication: `POST /api/register` does not exist. OIDC callback is the only entry point.) | README §A; ADR-0004; CON-003 | Security | Approved |
| CON-008 | **EN and AR are both first-class, not additive**: Full bilingual support and RTL layout from the first PH-1 release. Not a v2 retrofit. (Implication: Every UI component must be built with logical CSS from day 1. Every user-visible string must exist in both locale files before the feature ships. RTL regression must pass in CI.) | README §A; ADR-0012; P-07 | Product | Approved |
| CON-009 | **Keycloak is the sole IdP; roles sourced from Keycloak claims**: ACMP does not store or assign global roles. It reads claims on token validation and maps to canonical roles. (Implication: ACMP cannot have a "role management" screen that changes Keycloak data. Role assignment is done in Keycloak; ACMP reads the result.) | README §A; ADR-0004 | Security | Approved |
| CON-010 | **Single committee; no multi-committee generalization**: v1 is built for exactly one committee. No tenant IDs, no committee selection, no committee-scoped routing. (Implication: Reject any data-model change that adds a committee entity as a root tenant discriminator unless a new explicit decision reverses this.) | README §A (settled 2026-06-24) | Product | Approved |
| CON-011 | **Voting always attributed in v1; no anonymous voting**: Every voter's choice is recorded and visible to Auditor and Chairman. Anonymous voting mode is out of scope for v1. (Implication: The `anonymity` flag on the vote configuration (FR-070 references it) is an informational field only in v1; the system always records individual votes. Full anonymity support is a future decision.) | README §A (ADR-0010; settled 2026-06-24) | Security | Approved |
| CON-012 | **Gregorian dates only; no Hijri in v1**: All date storage, display, and SLA calculations use Gregorian calendar. (Implication: No Hijri calendar library, no dual-calendar display, no Hijri date input. Localized Gregorian formatting (right-to-left digit order in AR) is acceptable.) | README §A (settled 2026-06-24) | Product | Approved |
| CON-013 | **Keystone is optional**: The Research module must work fully standalone (manual entry). Keystone import is an enhancement. No hard dependency on Keystone availability. (Implication: Research Mission record can be created without a Keystone package reference. The Keystone import tool is a PH-2 feature, not a gate for the Research module.) | README §A; ADR-0007 | Product | Approved |
| CON-014 | **Webex is Phase 2, adapter-only**: No Webex calls in PH-1 code. All Webex integration is behind the `INotificationChannel` and `IWebexRecordingAdapter` abstractions. (Implication: PH-1 code must compile and run without Webex credentials configured. The adapter registers as no-op when Webex config is absent.) | README §A (settled 2026-06-24); ADR-0005 | Product | Approved |
| CON-015 | **AI extraction is Phase 3; manual always available**: No AI features in PH-1 or PH-2. When delivered in PH-3, AI extraction is off by default and requires Admin activation with data-residency confirmation. (Implication: No LLM client code in PH-1 or PH-2 production code. Phase 3 LLM integration is behind a feature flag.) | README §A (settled 2026-06-24) | Product | Approved |
